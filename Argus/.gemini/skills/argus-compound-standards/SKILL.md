# Argus-compound 项目技能规范

> **适用范围**：Argus-compound（24小时之眼）全部子模块  
> **版本**：v1.1 · 2026-02-11  
> **维护者**：项目核心团队

---

## 一、语言与表述规范

| 项目 | 要求 |
|------|------|
| **默认语言** | 所有交流、报告、文档说明均使用**中文** |
| **例外情况** | 代码注释、API 文档中的技术术语、变量/函数命名可使用英文 |
| **命名风格** | 文件名使用小写英文 + 短横线（kebab-case），中文标题写在文档内部 |

---

## 二、文档目录结构

所有文档统一存放于 `docs/` 目录下，子目录划分如下：

```text
docs/
├── image/                # 图片、截图、录屏等影像材料
├── gouji/                # 各模块架构报告（Architecture）
│   ├── go-sensory.md
│   ├── web-console.md
│   ├── py-vision.md      # [已归档] 功能已迁移至 go-sensory
│   └── openclaw-skills.md # [已归档] 功能已迁移至 go-sensory
└── renwu/                # 规划、计划、审计报告（Tasks & Audits）
    ├── plan-YYYYMMDD.md
    └── audit-YYYYMMDD.md
```

### 各目录用途

| 目录 | 用途 | 命名规则 |
|------|------|----------|
| `image/` | 存放截图、架构图、流程图、录屏等影像素材 | `{模块名}-{描述}.{ext}` 如 `py-vision-api-flow.png` |
| `gouji/` | 存放每个模块的**架构报告**，包括模块职责、目录结构、核心类/函数、数据流、依赖关系 | 以模块名命名，如 `go-sensory.md` |
| `renwu/` | 存放**规划计划**（plan）、**审计报告**（audit）、**优化方案**（optimize）等 | `{类型}-{日期}.md` 如 `plan-20260210.md` |

## 三、架构文档维护规范

> **核心原则：每完成一个板块的任务，必须同步更新或生成该板块的架构文档。**

### 架构文档 (`gouji/`) 标准模板

每份架构文档应包含以下章节：

```markdown
# {模块名} 架构文档

## 概述
模块定位、核心职责、一句话描述。

## 技术栈
语言、框架、关键依赖。

## 目录结构
```text
模块根目录树状图
```

## 核心组件

各核心文件/类/函数的职责说明。

## 数据流

请求/响应流或数据处理流程。

## 接口清单

对外暴露的 API / 函数接口。

## 配置说明

环境变量、配置文件说明。

## 变更记录

| 日期 | 变更内容 | 操作人 |
|------|----------|--------|

```

### 更新时机

- ✅ 新增/删除核心文件或模块时
- ✅ 修改 API 接口、数据模型时
- ✅ 重构目录结构或依赖关系时
- ✅ 完成阶段性任务交付时

## 四、代码规范

### 4.1 高内聚低耦合

| 原则 | 说明 |
|------|------|
| **高内聚** | 每个模块/文件应只负责一项明确职责，相关逻辑集中在一起 |
| **低耦合** | 模块间通过清晰的接口通信，避免直接依赖内部实现 |
| **禁止为拆分而拆分** | 不以文件数量为目标；只有当逻辑确实独立且可复用时才抽离 |

### 4.2 拆分标准

仅在满足以下**至少一项**条件时才进行拆分：

1. **职责独立** — 这段逻辑有独立的业务含义，可独立测试
2. **复用需求** — 已存在或即将存在 ≥2 处调用
3. **复杂度阈值** — 单文件超过 **300 行**，或单函数超过 **50 行**
4. **关注点分离** — 混合了路由/模型/业务/工具等不同层级的代码

### 4.3 语义化组件

- 组件/函数命名必须表达**意图**而非实现，如 `analyzeImage` 而非 `processData`
- 目录按**功能域**组织，非按文件类型（优先 `auth/` 而非 `controllers/`）
- 公共工具函数统一放在 `utils/` 或 `shared/`，不分散在各业务模块

### 4.4 各语言附加规范

| 语言 | 规范要点 |
|------|----------|
| **Go** | 使用 `internal/` 隐藏内部包；错误显式处理不忽略；接口定义在消费侧；CGO 文件可豁免行数限制 |
| **Python** | 类型注解覆盖所有公共函数；使用 Pydantic 做数据校验；避免循环导入 |
| **TypeScript** | 严格模式 (`strict: true`)；优先使用 `interface` 而非 `type`；组件 Props 必须类型化 |

## 五、任务规划规范

> **核心原则：避免上下文过载，拒绝一次性处理过多任务。**

### 5.1 任务拆分策略

```

大目标 → 阶段（Phase）→ 步骤（Step）→ 动作（Action）

```

| 层级 | 粒度 | 示例 |
|------|------|------|
| **阶段** | 1 个独立可交付的功能块 | "P1: 修复文件上传 Bug" |
| **步骤** | 1 次对话可完成的工作量 | "Step A: 修改后端上传接口" |
| **动作** | 1 个具体的编码或验证操作 | "修改 `upload.go` 的文件类型校验逻辑" |

### 5.2 单次对话工作量控制

| 指标 | 上限 | 原因 |
|------|------|------|
| 并行修改的文件数 | **≤ 5 个** | 防止上下文窗口溢出 |
| 单步骤涉及的模块数 | **≤ 2 个** | 跨模块改动需要逐步验证 |
| 单次 Plan 中的步骤数 | **≤ 4 个** | 保持计划可执行且可追踪 |

### 5.3 执行节奏

1. **计划先行** — 先输出本次任务的拆分计划，确认后再动手
2. **逐步推进** — 完成一个步骤后验证，再进入下一步
3. **及时归档** — 每个阶段完成后更新 `gouji/` 架构文档和 `renwu/` 任务报告
4. **断点可恢复** — 任务拆分应确保每个步骤完成后系统处于可运行状态

### 5.4 计划文档模板（`renwu/plan-YYYYMMDD.md`）

```markdown
# {任务名称} 执行计划

## 背景与目标
简述为什么做、做什么、期望结果。

## 阶段划分

### P1: {阶段名}
- **目标**：
- **步骤**：
  - [ ] Step A: ...
  - [ ] Step B: ...
- **验证方式**：
- **预计工作量**：{N} 次对话

### P2: {阶段名}
...

## 风险与注意事项
- ...

## 变更记录
| 日期 | 变更内容 |
|------|----------|
```

## 六、验证与质量保障

| 环节 | 要求 |
|------|------|
| **编译/构建** | 每次修改后确保 `go build` / `npm run build` / `python -m py_compile` 通过 |
| **功能验证** | 关键路径需通过浏览器或 API 工具实际验证 |
| **回归检查** | 修改公共模块时需确认上下游不受影响 |
| **文档同步** | 代码变更后 24 小时内同步更新对应架构文档 |

## 七、Git 提交规范

```text
<类型>(<作用域>): <简要描述>

类型：feat | fix | refactor | docs | chore | test
作用域：go-sensory | web-console | docs | agent | analysis | metrics | pipeline | skills | vlm
```

示例：

```text
feat(agent): 新增 ReAct 循环多步骤验证
docs(gouji): 更新 go-sensory 架构文档
fix(web-console): 修复仪表盘实时数据不刷新的问题
refactor(vlm): 提取 CRUD handlers 到 router_config.go
```
