## 🔴 优先级 P1 — 建议近期改进（Medium 风险）

### 1. Memory Sync 写锁粒度过粗

| 属性 | 值 |
|------|-----|
| **文件** | `backend/internal/memory/manager.go:296-297` |
| **报告** | audit-memory.md |
| **问题** | `Sync()` 方法全程持有 `mu.Lock()`（写锁），包括文件扫描、embedding 生成、DB 写入整个流程。大工作区同步耗时较长，期间所有 `Search()`/`ReadFile()` 调用被阻塞。 |
| **影响** | 用户在后台同步期间无法搜索记忆 |
| **建议修复** | 拆分为两阶段：<br>① 无锁阶段：文件扫描 + embedding 生成（CPU/IO 密集）<br>② 持锁阶段：仅 DB 写入（毫秒级）<br>`dirty` 列表构建和 embedding 生成均不修改 Manager 状态，可在锁外执行 |

```diff
 func (m *Manager) Sync(ctx context.Context, opts *SyncOptions) error {
-    m.mu.Lock()
-    defer m.mu.Unlock()
+    // Phase 1: 无锁 — 扫描 + embedding
+    files, dirty, embeddings := m.prepareSyncUnlocked(ctx, opts)
+
+    // Phase 2: 持锁 — 仅 DB 写入
+    m.mu.Lock()
+    defer m.mu.Unlock()
+    return m.commitSync(ctx, dirty, embeddings)
 }
```

### 2. Embedding HTTP 请求无超时

| 属性 | 值 |
|------|-----|
| **文件** | `backend/internal/memory/embeddings.go:248` |
| **报告** | audit-memory.md |
| **问题** | `embeddingHTTPPost()` 使用 `http.DefaultClient`，无连接超时、无读取超时。如果 embedding API（OpenAI/Gemini/Voyage）响应慢或无响应，goroutine 将无限阻塞。 |
| **影响** | 同步过程可能永久挂起 |
| **建议修复** | 创建带超时的专用 Client |

```go
var embeddingClient = &http.Client{
    Timeout: 30 * time.Second,
    Transport: &http.Transport{
        MaxIdleConns:        10,
        IdleConnTimeout:     90 * time.Second,
        TLSHandshakeTimeout: 10 * time.Second,
    },
}
```

### 3. 心跳活跃时间窗不支持跨午夜

| 属性 | 值 |
|------|-----|
| **文件** | `backend/internal/infra/heartbeat.go` |
| **报告** | audit-infra.md |
| **问题** | 活跃时间窗仅支持 `start < end` 格式（如 `09:00-18:00`）。不支持跨午夜场景（如 `22:00-06:00`），导致夜间值班时心跳不触发。 |
| **影响** | 夜间作业的 Agent 心跳失效 |
| **建议修复** | 当 `start > end` 时取反逻辑：`now >= start || now <= end` |

### 5. 频道规范化函数内动态编译正则

| 属性 | 值 |
|------|-----|
| **文件** | `backend/internal/channels/normalize.go:64,295-300,377` |
| **报告** | audit-channels.md |
| **问题** | 3 个函数内部使用 `regexp.MustCompile()`：<br>- `NormalizeTelegramMessagingTarget` (L64)<br>- `LooksLikeSignalTargetID` (L295-300, 3处)<br>- `LooksLikeIMessageTargetID` (L377)<br>同文件中 `tgTargetRe`/`tgNumericRe`(L74-75) 已正确预编译，风格不一致。 |
| **影响** | 消息路径上的无谓 CPU 开销 |
| **建议修复** | 全部提升为包级 `var` 预编译 |

```go
var (
    tgLinkRe      = regexp.MustCompile(`t\.me/...`)
    signalUuidRe  = regexp.MustCompile(`^[0-9a-f]{8}-...`)
    signalNumRe   = regexp.MustCompile(`^\+?\d{7,15}$`)
    imessageMailRe = regexp.MustCompile(`^[^@]+@[^@]+\.[^@]+$`)
)
```

### 6. Go/Rust 类型同步无自动化机制

| 属性 | 值 |
|------|-----|
| **文件** | `backend/pkg/types/` (35 Go files) vs `cli-rust/crates/oa-types/` |
| **报告** | audit-cmd-pkg-bridge-rust.md |
| **问题** | Go 和 Rust 两端的类型定义（config/channels/agents/tools 等）需要手工同步，无 protobuf/JSON Schema 等共享机制。 |
