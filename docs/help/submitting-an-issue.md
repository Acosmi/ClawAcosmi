## 提交 Issue

清晰、简洁的 Issue 有助于加速诊断和修复。对于 Bug、回归或功能缺口，请包含以下信息：

### 需要包含的内容

- [ ] 标题：区域 & 症状
- [ ] 最小复现步骤
- [ ] 期望行为 vs 实际行为
- [ ] 影响和严重程度
- [ ] 环境信息：操作系统、运行时版本、配置
- [ ] 证据：脱敏后的日志、截图（不含个人信息）
- [ ] 范围：新问题、回归还是长期存在
- [ ] 已搜索代码库和 GitHub 确认无重复 Issue
- [ ] 确认问题未在最近版本中修复（尤其是安全问题）
- [ ] 结论基于证据或可复现

保持简短。精炼 > 完美语法。

提交前验证（运行并修复失败项）：

- `cargo clippy` — Rust lint 检查
- `cargo test` — Rust 单元测试
- `go vet ./...` — Go 静态分析
- `go test ./...` — Go 单元测试
- `make build` — 完整构建

### 模板

#### Bug 报告

```md
- [ ] 最小复现
- [ ] 期望 vs 实际
- [ ] 环境信息
- [ ] 受影响的渠道、未受影响的渠道
- [ ] 日志/截图（已脱敏）
- [ ] 影响/严重程度
- [ ] 临时解决方案

### 概述

### 复现步骤

### 期望行为

### 实际行为

### 环境信息

### 日志/证据

### 影响

### 临时解决方案
```

#### 安全问题

```md
### 概述

### 影响

### 版本信息

### 复现步骤（可安全分享的部分）

### 缓解/临时方案

### 证据（已脱敏）
```

_避免在公开场合泄露密钥/漏洞细节。对于敏感问题，请尽量减少细节并申请私密披露。_


### 提交修复 PR

可先提 Issue 再提 PR，也可直接提 PR。如跳过 Issue，请在 PR 中包含详细信息。保持 PR 聚焦、注明 Issue 编号、添加测试或说明缺少测试的原因、记录行为变更/风险、附上脱敏后的日志/截图作为证据，并在提交前运行验证命令。
