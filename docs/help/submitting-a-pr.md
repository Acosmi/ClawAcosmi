> [!NOTE]
> 本文档已更新以适配 **Rust CLI + Go Gateway** 混合架构。

好的 PR 便于审查：审查者应该能快速理解意图、验证行为，并安全地合入变更。本指南涵盖面向人工和 LLM 审查的简洁、高信号提交要求。

## 优秀 PR 的标准

- [ ] 说明问题、重要性和变更内容
- [ ] 保持变更聚焦，避免大范围重构
- [ ] 总结用户可见/配置/默认值变更
- [ ] 列出测试覆盖范围、跳过项和原因
- [ ] 附上证据：日志、截图或录屏（UI/UX 相关）
- [ ] 提交前运行/修复相关验证命令
- [ ] 搜索代码库和 GitHub 查找相关功能/Issue/修复
- [ ] 结论基于证据或观察
- [ ] 好标题：动词 + 范围 + 结果（例如 `Docs: add PR and issue templates`）

简洁为上；简练的审查 > 完美语法。省略任何不适用的章节。

### 基线验证命令（提交前运行并修复失败项）

- `cargo clippy` — Rust lint 检查
- `cargo test` — Rust 单元测试
- `cargo build` — Rust 构建
- `go vet ./...` — Go 静态分析
- `go test ./...` — Go 单元测试
- `make build` — 完整项目构建
- 协议变更：`make protocol-check`

## 渐进式披露结构

- 顶部：概述/意图
- 其次：变更/风险
- 再次：测试/验证
- 最后：实现细节/证据

## 常见 PR 类型的具体要求

- [ ] 修复：添加复现步骤、根因分析、验证方式
- [ ] 功能：添加用例、行为描述/演示/截图（UI）
- [ ] 重构：声明"无行为变更"，列出移动/简化的内容
- [ ] 维护：说明原因（如构建耗时、CI、依赖）
- [ ] 文档：提供前后对比上下文，链接到更新的页面
- [ ] 测试：说明覆盖了什么缺口，如何防止回归
- [ ] 性能：附上前后指标和测量方法
- [ ] UX/UI：截图/视频，说明可访问性影响
- [ ] 基础设施/构建：环境/验证步骤
- [ ] 安全：总结风险、复现、验证，不含敏感数据。仅基于有据可查的结论

## 检查清单

- [ ] 清晰的问题/意图
- [ ] 聚焦的范围
- [ ] 列出行为变更
- [ ] 列出测试及结果
- [ ] 手动测试步骤（适用时）
- [ ] 无密钥/私人数据
- [ ] 基于证据
