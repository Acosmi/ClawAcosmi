## 【技能二：执行 SOP 工作流】

**角色**：标准化审计执行流程，确保有序推进。

**规则**：不要一上来就修改代码。对于目录内的核心文件，必须严格按以下步骤执行：

**第零步：联网可信源验证（新增）**
制定任何审计/补全计划前，必须通过联网查阅可信源（官方文档、GitHub issue、Go/Node.js 官方 spec 等）验证关键技术假设的正确性。

**第一步：静默提取与对比**
仔细阅读对应的 TS 和 Go 文件。强制检查以下"隐性依赖和逻辑映射"：
1. `process.env` 和隐蔽的闭包全局变量是否丢失？
2. TS 的隐式状态传递，是否在 Go 中被 `context.Context` 妥善接管？
3. TS 的 Falsy 判断 (`!val`)、可选链 (`?.`) 和 JSON `undefined` 忽略特性，Go 的 Struct (指针/`omitempty`) 是否完美兼容还原？
4. TS 的 `Promise/async` 转换为 Go 的 `goroutine` 后是否存在并发泄漏或死锁风险？错误捕获 (`try-catch` vs `err != nil`) 是否等价且未丢失上下文？

**第二步：输出《深度审计与差异报告》**
打印 Markdown 报告，清晰列出：
- 完美对齐的逻辑
- Go 代码中遗漏的显性/隐性逻辑
- 外部依赖（NPM）的能力缺失提示

**第三步：阻断等待**
在报告末尾附上建议修改/补全的 Go 代码草稿，然后**停止操作**，等待用户指令。

**第四步：精准补全**
只有在用户明确回复"同意修复"或"同意补全"后，才能调用编辑工具进行局部代码写入。

## 【技能三：复核审计技能 (Post-Task Review Audit)】

**角色**：交叉验证，确保移植零遗漏。

**规则**：当某个重构任务初步完成，或任何文件准备归档前，强制触发本技能：
1. **交叉颗粒度审计**：对原 TS 代码与现有 Go 代码进行"代码层级颗粒度"的逐行/逐逻辑块交叉比对。
2. **审计重点雷达**：死磕【显性依赖】和【隐藏依赖】（如环境变量穿透、上下文状态、闭包、并发模型差异等）是否 100% 继承或安全平替。
3. **归档红线校验 (绝对规则)**：所有需归档的文件，必须严格按照文件中记录的代码文件清单，全部执行完此交叉颗粒度复审。**复核审计未通过，绝对禁止任何归档操作！**
4. **触发下游**：只有复核审计彻底通过后，才能触发【技能四】中的"更新文档链条"与归档流程。
