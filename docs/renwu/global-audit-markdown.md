### 架构演进与设计选择

本模块在跨语言重构时，做出了**最关键的核心架构分化**，这主要体现在底层解析器的选择上：

1. **纯正内部解析器 (Bespoke Parser)**: TS 侧严重依赖了大型第三方包 `markdown-it` 来将 Markdown 文字转化为抽象语法树 (Token stream)，而后再组装生成自有的 `MarkdownIR`。在 Go 重构中，为了极致追求轻量化和极快的执行速度，**放弃了引入 `yuin/goldmark` 等任何第三方重量级 AST 解析器，手写了一个基于正则和简单状态机的 `parseMarkdownSimple`。**
2. **栈式渲染器 (Render.ts/go)**: 这部分用于将 `MarkdownIR` 重新渲染回纯文本，并在边界处插入控制符/ANSI 颜色，算法原理上维持了 1:1 (LIFO 栈式关闭策略重叠边界)。

## 差异清单

### P3 架构分化：表格转换的解耦

| ID | 描述 | TS 实现 | Go 实现 | 修复建议 |
|----|------|---------|---------|----------|
| MD-1 | **Table 模式的转化时机** | 内聚在 `ir.ts` 中。由于 `markdown-it` 能够天然解析 `table`、`th`、`tr` Token。TS 代码在遍历 Token 树时顺手将表格转为 Bullets 或对齐的 Code block。 | `tables.go` 被彻底解耦。由于 `parseMarkdownSimple` 无法处理表格，Go 选择了 **通过正则在原始 string 级别直接提取表格边界**，在转为 IR 之前就将 Markdown 原文里的表格强行文本替换为了 Bullets/Code，然后再交给简单解析器处理。 | **优秀的设计优化 (P3/提升)**。解耦做得很好，无需修复。 |

### P3 细微差异：边缘语法支持

| ID | 描述 | TS 实现 | Go 实现 | 修复建议 |
|----|------|---------|---------|----------|
| MD-2 | **边缘 Markdown 语法** | `markdown-it` 支持各种深层嵌套语法（例如嵌套列表包引用块包表格）。 | Go 版基础解析器对复杂的深层嵌套列表可能有解析降级。 | 对于 CLI 工具核心用例（聊天对话渲染、代码块截断），Go 版足够强壮。无需修复。 |
| MD-3 | **Frontmatter 解析** | 使用 `@std/yaml` 或自带正则解析。 | 结合 `gopkg.in/yaml.v3` 并行混查，提取 YAML 块或 `key: value` 模式。1:1 对齐。 | 无需修复。 |
