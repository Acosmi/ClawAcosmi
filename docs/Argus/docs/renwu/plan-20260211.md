### P2: 关键代码拆分（文件 ≥400 行 + 函数 ≥100 行）

> **原则**：§4.2 "单文件超过 300 行" + "禁止为拆分而拆分"。仅对严重超标且有明确职责边界的代码拆分。

- **步骤**：

    提取 `initCapture()`、`initVLM()`、`initPipeline()`、`printBanner()`、`waitForShutdown()` 等子函数。保持单文件，但函数粒度合规
  - [ ] Step B: `vlm/router.go`（433 行）  
    提取 CRUD handlers → `vlm/router_config.go`（handleConfigProviders/handleCreate/Update/Delete/SetActive/List ~150 行）

## 风险与注意事项

| 风险 | 影响 | 缓解措施 |
| --- | --- | --- |
| CGO 文件拆分受限 | `darwin_sck.go` 无法完全达标 | 标注"CGO 豁免"并在架构文档说明 |
| 接口重定义可能引入冗余 | 同一实体多处接口定义 | Go 惯用法天然支持，用注释标注关系 |
| 函数拆分可能降低可读性 | 过度拆分反而增加认知负担 | §4.1 "禁止为拆分而拆分"，保留内聚逻辑 |
| 文档更新时效性 | 后续代码变更可能导致文档再次过时 | P1 完成后建立 checklist，每次 PR 检查 |

## 执行优先级

```text
P1 (架构文档) → P2 (关键拆分) → P3 (中等拆分) → P4 (接口/规范)
```

每阶段完成后 `go build` + `go vet` 验证，确保系统始终可运行。
