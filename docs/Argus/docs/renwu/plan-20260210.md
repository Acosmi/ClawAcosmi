# Argus-compound 架构演进计划：Go 统一后端 + Ollama 模型层

> 日期：2026-02-10  
> 状态：待执行  
> 关联审计：i18n 全局审计已完成

## 背景与目标

当前 Argus-compound 采用 Go + Python 混合架构。`py-vision` 模块依赖 FastAPI/Pillow/numpy 等 Python 生态，但审计发现其核心功能——VLM API 代理、图像预处理、REST 服务——Go 原生均可胜任。

**目标**：消除 Python 依赖，Go 统一后端，引入 Ollama 作为本地/云端模型推理层，新增模型配置管理 API。

**架构演进路线**：

```
┌──────────────┐
│  web-console │  Next.js 前端（:3090）
└──────┬───────┘
       │ HTTP / WS
┌──────▼───────────────────────────────────┐
│             go-sensory (:8090)            │
│  统一网关 + 传感器 + VLM路由 + 配置管理    │
│  ┌─────────┐ ┌──────────┐ ┌───────────┐  │
│  │屏幕采集 │ │模型路由器│ │配置管理API│  │
│  │SCK / CG │ │/v1/chat  │ │/api/config│  │
│  └─────────┘ └────┬─────┘ └───────────┘  │
└────────────────────┼─────────────────────┘
                     │ HTTP (OpenAI 兼容)
          ┌──────────┴──────────┐
          ▼                     ▼
  ┌───────────────┐    ┌───────────────┐
  │ Ollama        │    │ 云端 API       │
  │ 本地/局域网/云 │    │ OpenAI/Gemini │
  │ llama.cpp C++ │    │ Claude 等     │
  └───────────────┘    └───────────────┘
```

## 关于 C++ CGO 的定位

经过调研，CGO 调用 C++ 的真正价值在于**进程内高性能推理**（ONNX Runtime / TensorRT）和**高性能图像管线**（OpenCV C++）。Ollama 底层已经是 llama.cpp（纯 C++），通过 HTTP API 调用等效于间接使用 C++ 推理引擎。

> **结论**：现阶段通过 Ollama HTTP 调用即可，CGO C++ 层作为 P3 远期预留，在需要边缘推理/GPU 自定义算子时再引入。

## 阶段划分

### P1: Go 吞并 py-vision 核心功能

- **目标**：在 Go 层实现 VLM 代理和模型路由，消除 py-vision 运行时依赖
- **步骤**：
  - [x] Step A: 创建 `internal/vlm/` 模块 — 定义 Provider 接口 + OpenAI 兼容 HTTP client
  - [x] Step B: 实现模型路由器 — 根据 provider 配置转发请求到不同后端
  - [x] Step C: 迁移图像预处理 — Go 原生 `image/jpeg` + `encoding/base64` 替代 Pillow
  - [x] Step D: web-console API 代理 — `next.config.js` 添加 rewrites 解决跨端口问题
- **验证方式**：通过 web-console 调用 `/v1/chat/completions`，请求成功转发到云端 API
- **预计工作量**：2 次对话

---

### P2: 模型配置管理 API + 前端对接

- **目标**：新增配置 CRUD 接口，web-console 设置页接真实数据
- **步骤**：
  - [x] Step A: 创建 `internal/config/` 模块 — Provider CRUD + JSON 持久化
  - [x] Step B: 注册 REST 路由 — `GET/POST/PUT/DELETE /api/config/providers`
  - [x] Step C: 前端对接 — SettingsPage 读写配置 API，替换硬编码数据
  - [x] Step D: 添加 provider 健康检查 — 定时 ping 各后端，前端显示状态
- **验证方式**：前端设置页可增删 provider，切换 active provider，状态实时更新
- **预计工作量**：2 次对话

---

### P3: Ollama 独立部署集成（架构变更）

- **目标**：Ollama 作为独立服务部署，供 go-sensory 和视觉理解智能体共同使用
- **架构决策**：
  - Ollama 独立部署运行，不再嵌入 go-sensory
  - Ollama 暴露 OpenAI 兼容 API（`http://<host>:11434/v1`）
  - go-sensory 通过已有的 `OpenAIProvider` 直接对接，无需专用 Ollama Provider
  - 视觉理解智能体也通过 OpenAI 兼容协议连接同一 Ollama 实例
- **当前状态**：✅ 已被 P1 + P2 覆盖
  - [x] OpenAI 兼容 Provider 已支持任意端点（包括 Ollama）
  - [x] 前端可通过设置页添加 Ollama 类型 provider（type=openai，endpoint 指向 Ollama）
  - [x] 健康检查自动对 Ollama 端点进行 ping
- **配置示例**：

  ```json
  {
    "name": "ollama-local",
    "type": "openai",
    "endpoint": "http://localhost:11434/v1",
    "api_key": "",
    "model": "qwen2.5-vl:7b",
    "active": true
  }
  ```

---

### P4: C++ 高性能推理层（远期暂缓）

- **目标**：为边缘推理 / 离线场景预留 CGO C++ 通道
- **当前状态**：⏸ 暂缓执行，前期 Ollama 满足业务需求
- **触发条件**（满足任一即启动）：
  - 需要进程内 ONNX Runtime 推理（不走 Ollama）
  - 需要 OpenCV C++ 高性能图像管线
  - 需要 GPU 自定义算子
- **步骤**：
  - [ ] Step A: 创建 `internal/cppbridge/` — C bridge 函数 + Go wrapper
  - [ ] Step B: 集成 ONNX Runtime C++ — 模型加载 + 推理接口
  - [ ] Step C: 静态链接打包 — 确保单一二进制部署
- **预计工作量**：3+ 次对话

## 风险与注意事项

| 风险 | 缓解措施 |
|------|----------|
| Go HTTP client 不支持某些 VLM API 特殊格式 | 以 OpenAI 兼容协议为标准，各 provider adapter 处理差异 |
| Ollama 连接不稳定（云端部署） | provider 健康检查 + 自动 fallback 到备用 provider |
| P1 迁移期间 py-vision 功能回退 | 保留 py-vision 代码不删除，直到 P2 验证完成 |
| CGO 引入后交叉编译受限 | P4 做为独立 build tag，默认不启用 |
