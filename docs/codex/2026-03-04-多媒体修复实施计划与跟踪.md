- 关联审计：`docs/codex/2026-03-04-多媒体全链路审计报告.md`

## 1. 目标与完成定义

目标：在不破坏现有会话与渠道能力的前提下，完成多媒体链路的可用性、稳定性、一致性治理。

完成定义（DoD）：

1. 配置修改后无需重启可即时生效。
2. DocConv 测试与执行结果一致，无“假通过”。
3. 入站和出站都支持多媒体数组，不再仅保留第一项。
4. 上传限制与错误提示在前端/网关/`send`/渠道一致。
5. DingTalk/WeCom 具备多模态最小可用链路。
6. 关键链路具备自动化回归测试。

## 4. 关键实施细节

## 4.1 MM-001 运行态热刷新（P0）

- 改造点：
  - 将飞书 `DispatchMultimodalFunc` 从“启动期闭包 Provider”改为“运行时读取最新配置并获取 Provider”。
  - 增加轻量缓存（例如 TTL 30~60s）避免每条消息全量重建。
- 验收：
  - 修改 STT/DocConv/Image 后，下一条飞书消息立即生效。
  - 无需重启后端服务。

## 4.2 MM-002 DocConv 一致性修复（P0）

- 改造点：
  - `docconv.test` 必须反映真实执行能力。
  - 若 `sse` 当前不支持 convert，测试明确失败并给出说明。
- 验收：
  - 不再出现“测试通过但执行失败”的假阳性。

## 4.3 MM-003 + MM-014 多媒体结构升级（P0/P1）

- 改造点：
  - 从单 `MediaData/MediaBase64` 升级到 `[]MediaItem`。
  - `PreprocessResult` 同步升级，入站多图不再丢首图外数据。
  - 提供兼容层，避免旧调用路径一次性全断。
- 验收：
  - 智能体返回 2+ 图片可完整下发。
  - 飞书入站 2+ 图片可完整透传到前端显示层。

## 4.4 MM-005 + MM-015 限制统一（P1）

- 改造点：
  - 建立统一限制常量（单源配置），覆盖前端/网关/`send`/渠道。
  - 对飞书下载/上传限制窗口做统一策略（同上限或显式中间态错误提示）。
  - 前端禁止静默丢弃，改为显式提示。
- 验收：
  - 同一文件在 WebChat、`chat.send`、`send`、渠道行为一致。

## 4.5 MM-013 渠道接入补齐（P1）

- 改造点：
  - 网关为 DingTalk/WeCom 注入 `DispatchMultimodalFunc`。
  - 钉钉/企微入站处理新增附件抽取。
  - 出站支持消费 `DispatchReply` 媒体字段并发送。
- 验收：
  - DingTalk/WeCom 至少实现图/音/文档中的已支持子集，并通过回归。

## 4.6 MM-016 Provider 生命周期优化（P2）

- 改造点：
  - `chat.send` 附件处理路径引入 Provider 复用/缓存。
  - 统一“刷新 vs 复用”策略，避免飞书与 webchat 的极端差异。
- 验收：
  - 高频附件场景 CPU/延迟优于基线，功能行为保持一致。

## 4.7 MM-009 测试体系补齐（P2）

- 用例最小集：
  - 入站音频 -> STT -> 回复。
  - 入站文档 -> DocConv -> 回复。
  - 入站多图 -> 多图透传 -> 前端显示。
  - 智能体多图产出 -> 渠道多图发送。
  - 超限文件 -> 一致错误码与文案。
  - DingTalk/WeCom 多模态最小链路冒烟。
- 验收：
  - CI 新增多媒体回归任务并纳入发布门禁。
