## 1. 执行摘要

本次审计聚焦“全局聊天附件上传、图片/语音/文档解析、智能体产出媒体转发到飞书等渠道”的端到端链路。修订后结论如下：

- 发现 **3 个 P0（阻断级）**、**7 个 P1（高优先级）**、**3 个 P2（优化级）**。
- 你反馈的核心症状（配置后不生效、文档转 mk 不可用、媒体链路不稳定）均有明确代码级根因。
- 关键问题不是单点，而是“**运行态刷新机制 + DocConv 兼容语义 + 单媒体数据结构限制 + 渠道实现不一致**”叠加。

## 2. 审计范围

- 前端：聊天附件输入、上传限制、多媒体配置向导文案与交互。
- 网关：`chat.send` 附件处理、飞书多模态预处理、STT/DocConv/Image Provider 初始化与调用。
- 媒体层：STT 本地实现、DocConv builtin/MCP 实现。
- 渠道层：飞书、钉钉、企微入站解析与出站发送。

## 3. 当前链路概览（简化）

1. 前端附件采集（文件/拖拽/粘贴）
2. `chat.send` 发送 base64 附件到网关
3. 网关按类型调用 STT/DocConv/Image 预处理
4. Agent 执行并产生文本/媒体回复
5. 渠道插件上传并回发文本/图片/文件

## 4. 发现清单

## P0（阻断级）

### P0-01 飞书多媒体 Provider 仅在启动时初始化，配置修改后不热生效

- 现象：多媒体配置保存成功，但飞书语音/文档仍报未配置或走旧配置。
- 根因：`feishuPreprocessor` 在网关启动阶段构造并闭包复用，未在配置变更后刷新。
- 证据：
  - `backend/internal/gateway/server.go:2072-2104`
  - `backend/internal/gateway/server_methods_stt.go:114-125`
  - `backend/internal/gateway/server_methods_docconv.go:132-143`
  - `backend/internal/gateway/server_methods_image.go:119-130`
- 影响：必须重启后端才“看似恢复”，前端重启无效。

### P0-02 DocConv 测试与执行能力不一致，存在“测试通过但实际不可用”

- 现象：DocConv 向导测试通过，真实转换失败。
- 根因：`docconv.test` 对 `sse` 仅校验 URL 非空；`Convert` 却强制仅支持 `stdio`。
- 证据：
  - `backend/internal/media/docconv_mcp.go:92-94`
  - `backend/internal/media/docconv_mcp.go:150-155`
  - `ui/src/ui/views/wizard-docconv.ts:268-285`
- 影响：用户判断被误导，配置体验与运行行为严重偏离。

### P0-03 回复媒体结构是单媒体，导致多媒体回复被截断

- 现象：智能体多图/多附件产出时，仅首个媒体被转发。
- 根因：链路数据结构为单 `MediaData/MediaBase64`，提取函数也只取第一个。
- 证据：
  - `backend/internal/channels/channel_message.go:50-54`
  - `backend/internal/gateway/dispatch_inbound.go:160-168`
  - `backend/internal/gateway/server_methods_chat.go:594-596`
  - `backend/internal/gateway/server.go:1928-1932`
- 影响：多媒体能力上限被结构性限制，不满足全链路目标。

## P1（高优先级）

### P1-01 DocConv MCP 未复用项目已有 MCP 生命周期客户端

- 现象：兼容性不稳定，部分 MCP 服务调用失败。
- 根因：DocConv 路径直接拼 JSON-RPC 调 `tools/call`，未标准握手。
- 对照证据：
  - DocConv 实现：`backend/internal/media/docconv_mcp.go:72-103`
  - 标准握手实现：`backend/internal/mcpclient/client.go:143-177`
  - 远程握手实现：`backend/pkg/mcpremote/client.go:85-121`
- 影响：跨 MCP 服务可用性和可维护性偏弱。

### P1-02 PDF 转 Markdown 能力声明与实际可用性存在偏差风险

- 现象：用户反馈“文档转 mk 不可用”。
- 根因：builtin 路径将 `.pdf` 映射为 pandoc 输入格式 `pdf`，并在文案中强调支持范围，实际行为易失配。
- 证据：
  - `backend/internal/media/docconv_builtin.go:160-163`
  - `backend/internal/media/docconv_builtin.go:46-48`
  - `ui/src/ui/locales/zh.ts:1717`
- 影响：用户预期与实际偏差大，失败率高。

### P1-03 上传大小限制体系不一致（四层限制 + 资源窗口不一致）

- 现象：同一文件在不同入口表现不一致。
- 根因：
  - 前端附件 10MB 且超限直接跳过无提示
  - STT 转写接口 25MB
  - `send` 方法 `mediaBase64` 解码后 10MB 限制
  - 飞书文件上传 30MB
  - 飞书资源下载 50MB，与上传 30MB 形成 30~50MB 行为窗口
- 证据：
  - `ui/src/ui/views/chat.ts:163-170`
  - `backend/internal/gateway/server_methods_stt.go:188-206`
  - `backend/internal/gateway/server_methods_send.go:147-155`
  - `backend/internal/channels/feishu/resource.go:70-77`
  - `backend/internal/channels/feishu/resource.go:178-181`
- 影响：用户感知“配置错误/随机失败/某些路径可用某些不可用”。

### P1-04 临时文件命名基于 PID，存在并发覆盖风险

- 现象：高并发下偶发转换异常或串数据。
- 根因：临时文件名仅使用 `pid`，并发请求可能重名。
- 证据：
  - `backend/internal/media/docconv_mcp.go:62,69`
  - `backend/internal/media/docconv_builtin.go:121`
  - `backend/internal/media/stt_local.go:49`
- 影响：稳定性与可重复性下降。

### P1-05 错误可观测性不足，初始化失败常被误判为“未配置”

- 现象：用户只能看到“STT 未配置”，但真实可能是二进制缺失/权限/初始化失败。
- 根因：初始化失败仅日志告警，用户态文本未做错误分层。
- 证据：
  - `backend/internal/gateway/server.go:2078,2086,2094`
  - `backend/internal/gateway/server_multimodal.go:155-157`
- 影响：定位慢、复现慢、支持成本高。

### P1-06 DingTalk / WeCom 多模态接入不完整（网关注入缺失 + 插件文本化处理）

- 现象：飞书具备多模态预处理路径，钉钉/企微在网关层仅注入 `DispatchFunc(text)`。
- 根因：
  - 网关未给钉钉/企微注入 `DispatchMultimodalFunc`
  - 钉钉回调仅取文本 `data.Text.Content`
  - 企微回调只上传 `msgType + content`，未抽取附件实体
  - 两插件发送逻辑目前主要发送文本，未消费 `DispatchReply` 媒体字段
- 证据：
  - `backend/internal/gateway/server.go:2137-2240`
  - `backend/internal/gateway/server.go:2246-2349`
  - `backend/internal/channels/dingtalk/handler.go:12-18`
  - `backend/internal/channels/wecom/handler.go:76-78,196-198`
  - `backend/internal/channels/dingtalk/plugin.go:119-135`
  - `backend/internal/channels/wecom/plugin.go:121-129`
- 影响：钉钉/企微多媒体链路不完整，用户感知为“配置已完成但渠道不可用”。

### P1-07 入站预处理结果仍是单图结构（PreprocessResult）

- 现象：飞书入站多图时，文本描述可拼接，但 base64 仅保留首图。
- 根因：`PreprocessResult` 只有 `ImageBase64/ImageMimeType` 单值字段，汇总阶段仅取首个。
- 证据：
  - `backend/internal/gateway/server_multimodal.go:33-41`
  - `backend/internal/gateway/server_multimodal.go:200-204`
- 影响：与 P0-03 同源，入站方向也存在多图截断。

## P2（优化级）

### P2-01 macOS 升级后的本地依赖路径变化缺少自愈与引导

- 现象：系统升级后 local-whisper 可能不可用。
- 根因：依赖路径验证严格但缺少自动探测/回填建议。
- 证据：
  - `backend/internal/media/stt_local.go:83-104`
  - `ui/src/ui/views/wizard-stt.ts:313,420`
- 影响：升级后可用性恢复依赖人工排障。

### P2-02 多媒体链路缺少完整集成回归测试

- 现象：回归风险高，功能演进易引入链路断点。
- 证据：
  - 现有相关测试集中在局部（如 `stt_dashscope_test.go`），未覆盖多媒体端到端。
- 影响：修复后难以防止二次回归。

### P2-03 `chat.send` 附件处理每次创建新 Provider 实例，存在性能浪费

- 现象：每条附件消息都会 `NewSTTProvider/NewDocConverter`。
- 根因：`chat.send` 路径按消息级重建 Provider，无复用策略。
- 证据：
  - `backend/internal/gateway/server_methods_chat.go:895-896`
  - `backend/internal/gateway/server_methods_chat.go:932`
- 影响：高频场景额外开销，尤其在 DocConv/MCP 下更明显；与 P0-01 的“飞书不刷新”形成策略不一致。

## 5. 根因归类

- 架构类：运行态配置未热刷新（P0-01）、单媒体数据结构（P0-03/P1-07）、渠道接入策略不一致（P1-06）。
- 一致性类：测试语义与执行语义不一致（P0-02）、多层限制不一致（P1-03）。
- 工程质量类：临时文件并发安全不足（P1-04）、错误分层不足（P1-05）、Provider 生命周期策略不一致（P2-03）、测试覆盖不足（P2-02）。

## 6. 修复优先顺序建议

1. P0-01 运行态热刷新
2. P0-02 DocConv 测试/执行语义对齐
3. P0-03 + P1-07 单媒体升级为多媒体数组（出站与入站一起改）
4. P1-03 限制统一（四层）+ 错误提示统一
5. P1-06 渠道接入补齐（DingTalk/WeCom 多模态）
6. P1-01 MCP 客户端统一化
7. P1-04/05 并发安全与错误分层
8. P2-01/02/03 体验、测试与性能治理

## 7. 验收基线（建议）

- 修改 STT/DocConv/Image 配置后，飞书入站无需重启即时生效。
- DocConv 任一 transport 的测试结果必须与实际执行一致。
- 智能体返回 2+ 图片时，渠道侧可完整发送（或明确上限并提示）。
- 飞书入站 2+ 图片时，前端可收到完整图片集合而非仅首图。
- 大小限制规则在前端、网关、`send` 方法、渠道层一致且有统一错误文案。
- DingTalk/WeCom 至少实现与飞书一致的多模态最小能力（图/音/文档中的已支持子集）。
- macOS 升级后可通过向导快速诊断并恢复本地依赖。
