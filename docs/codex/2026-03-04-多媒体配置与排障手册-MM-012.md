# 多媒体配置与排障手册（MM-012）

更新时间：2026-03-04

## 1. 目标

- 统一“聊天附件上传、图片解析、语音解析、渠道回传”配置路径。
- 明确渠道能力差异，避免“已配置但不可用”的误判。

## 2. 配置清单（后端）

- STT：语音转文本 Provider（用于语音消息与语音附件解析）。
- DocConv：文档转换 Provider（用于文档转 markdown）。
- ImageUnderstanding：图片理解 Provider（主模型不支持多模态时兜底）。
- 多模态灰度开关：`OPENACOSMI_MULTIMODAL_CHANNELS`
  - `all/*/empty`：全开
  - `none/off/false/disabled/0`：全关（回滚到文本）
  - `feishu,dingtalk,wecom`：按渠道灰度

## 3. 能力矩阵（当前）

- Feishu：入站多模态解析 + 出站文本/媒体发送（完整链路）。
- DingTalk：入站多模态解析 + 出站媒体（含上传链路，失败文本降级）。
- WeCom：入站多模态解析 + 出站媒体发送。
- WebChat：`mediaItems[]` 与旧 `mediaBase64` 兼容消费。

## 4. 关键行为约束

- 回复结构采用数组优先：
  - 新字段：`mediaItems[]`（支持多图/多媒体）
  - 兼容字段：`mediaBase64/mediaMimeType`（保留第一优先消费能力）
- 统一限制策略：
  - 前端、网关、`chat.send`、渠道侧限制保持一致，避免“某路径可发、某路径拒绝”。

## 5. 常见问题与处理

### 5.1 “多图只显示一张”

- 检查返回 payload 是否包含 `mediaItems[]`；
- 若仅存在 `mediaBase64`，说明链路仍走旧字段，需排查中间转换节点。

### 5.2 “配置了 STT 仍提示未配置”

- 先确认配置是否热生效（无需重启）；
- 查看多媒体预处理日志，确认 provider 初始化是否失败；
- 若渠道受灰度开关限制，先检查 `OPENACOSMI_MULTIMODAL_CHANNELS`。

### 5.3 “系统升级后视觉能力失效”

- 参考：`2026-03-04-macOS升级后视觉子智能体恢复指引-MM-008.md`；
- 先跑 `argus.permission.check`，按恢复提示重新授权。

## 6. 回滚与灰度

1. 临时回滚：`OPENACOSMI_MULTIMODAL_CHANNELS=none`。  
2. 小流量恢复：`OPENACOSMI_MULTIMODAL_CHANNELS=feishu`。  
3. 渐进放量：`feishu,dingtalk` -> `all`。
