## 1. 本轮复核新增确认问题

### F-01（P1）钉钉入站回调路径未真正回传 DispatchReply 媒体

- 现象：`DispatchReply` 含媒体时，钉钉回调路径默认走文本降级提示，不尝试发送媒体。
- 根因：`DingTalkPlugin.Start` 的回调仅拼接 `dingtalkMediaFallbackFromDispatchReply`，未调用二进制媒体发送链路。
- 影响：看似“多模态已接入”，但钉钉回包仍以文本为主，媒体能力实际未生效。

### F-02（P2）钉钉降级文案语义过时

- 现象：降级提示写成“当前钉钉通道暂不支持自动回传媒体”。
- 根因：文案仍保留旧语义，与现有上传+发送能力不一致。
- 影响：误导排障方向，用户会误判为“功能未实现”而非“本次发送失败/类型不支持”。

### F-03（P2）企微媒体下载上限硬编码，未复用统一限制常量

- 现象：`wecom/client.go` 使用字面量 `30 * 1024 * 1024`。
- 根因：未复用 `channels.ChatAttachmentFileMaxBytes`。
- 影响：后续若调整统一限制，企微分支容易漏改。

## 2. 全量修复计划（系统实际版）

1. P0/P1 主链路收口
   - 持续保证飞书/钉钉/企微在入站预处理、出站发送、错误透传三条链路行为一致。
   - 对“能力存在但未调用”的路径优先补齐（本次先修钉钉入站媒体回传）。
2. 限制与错误语义一致化
   - 所有媒体大小阈值统一到 `channels` 常量体系。
   - 降级提示统一为“失败原因/可恢复动作”语义，避免“能力不存在”误导。
3. 管道级回归门禁
   - 保持 `make test-multimedia` 为主门禁。
   - 补齐新增逻辑的最小单测，避免回归。
4. 文档与执行追踪
   - 当天修复当天回写 `docs/codex` 跟踪，确保“审计-修复-验证”闭环可追溯。

### R-05 非 send/poll/send_media 的 outbound 入口接入统一 SendError 映射（完成）

- 入口：`security.remoteApproval.test`
- 文件：
  - `backend/internal/gateway/remote_approval.go`
  - `backend/internal/gateway/server_methods_remote_approval.go`
- 改动：
  - `RemoteApprovalNotifier.TestProvider` 改为返回结构化 `SendError`：
