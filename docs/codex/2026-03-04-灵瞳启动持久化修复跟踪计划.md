# 灵瞳（Argus）启动持久化修复跟踪计划

- 日期：2026-03-04
- 状态：Draft（待执行）
- 范围：视觉子智能体重启后无法启动（`argus [resolve]: binary_not_found`）
- 目标：消除“重启后失效/升级后复发”，建立可持续运维与发布链路

---

## 1. 根因分析（RCA）

### RC-01 路径发现不命中（主因）

- 现有解析链路为：`$ARGUS_BINARY_PATH` → `.app`候选路径 → `~/.openacosmi/bin/argus-sensory` → `PATH`。
- 本机编译或构建好的可执行文件可能落位在诸如 `Argus/build/` 或其他未配置的目录，不在当前的 `.app`候选白名单内。
- 结果：启动在 `resolve` 阶段即失败并抛 `binary_not_found`。

代码证据：

- `backend/internal/gateway/boot.go:488` (resolveArgusBinaryPath 真实入口)
- `backend/internal/gateway/boot.go:501` (resolveArgusBinaryPathWithError 完整链路及 $ARGUS_BINARY_PATH 检查)
- `backend/internal/argus/codesign_darwin.go:37` (FindAppBundleBinary)
- `backend/internal/argus/codesign_darwin.go:50` (buildAppBundleCandidates)

### RC-02 启动环境不稳定（重启放大因子）

- 当前 `ARGUS_BINARY_PATH` 未持久化，`~/.openacosmi/bin/argus-sensory` 也不存在。
- 后台/守护进程启动时，不能依赖交互 shell 的 PATH。

### RC-03 缺少显式 binaryPath 配置能力（设计缺口）

- `subAgents.screenObserver` 当前没有 `binaryPath` 字段，无法在配置层绑定稳定可执行路径。
- 导致 resolver 只能“猜路径”，可维护性弱。

代码证据：

- `backend/pkg/types/types_openacosmi.go:105`

### RC-04 签名信任链存在风险（次因）

- 已有打包 app，但当前检测到 `CSSMERR_TP_NOT_TRUSTED` 风险信号。
- 即使路径修复，后续 Gatekeeper/TCC 持久性仍可能受影响。

### RC-05 macOS 升级后的 TCC 重判（环境相关）

- 升级后屏幕录制/辅助功能授权可能失效。
- 系统需具备“可重授权 + 可重试”的恢复机制，而非手工排障。

---

## 2. 修复目标与成功标准

### 总目标

- 冷启动、重启、系统升级后三类场景均可恢复启动。
- 失败信息可定位到具体阶段（resolve/permission/start/handshake）。
- 不再依赖临时 shell 环境变量人工修复。

### 成功标准（DoD）

1. 连续重启 5 次，Argus 均可拉起。
2. 前端/接口可返回结构化 `phase + reason + recovery`。
3. 具备一键诊断与标准恢复流程。
4. 文档与代码解析规则一致。

## 3. 分阶段实施计划（含优先级）

| ID | 优先级 | 任务 | 具体动作 | 验收标准 |
| --- | --- | --- | --- | --- |
| ARGUS-001 | P0 | 立即止血 | 持久化 `ARGUS_BINARY_PATH`（写入 `~/.openacosmi/openacosmi.json` 的 `env.vars`）；同步准备 `~/.openacosmi/bin/argus-sensory` 兜底软链/拷贝 | 重启后可用，不依赖临时 export |

| ARGUS-004 | P1 | 一键诊断 | 新增 `argus.diagnose` RPC，输出 resolvedPath/exists/executable/codesign/TCC/recovery | 前端可直观看根因 |
| ARGUS-005 | P1 | 启动机制固化 | 后台启动统一绝对路径，不依赖隐式 PATH | “手工可用、重启失效”消失 |

| ARGUS-007 | P1 | 安装位标准化 | 规范安装位（`/Applications/Argus.app` 或 `~/.openacosmi/Argus.app`）并同步到 `~/.openacosmi/bin` | 解析路径可预测 |
| ARGUS-008 | P1 | 签名发布双轨 | 内部自用（本地可运行）与发行模式（Developer ID + notarization）双轨文档化 | 自用与分发均有标准流程 |
| ARGUS-009 | P2 | 回归测试补齐 | 增加 resolver、重启场景、TCC/权限恢复回归测试 | CI 覆盖关键故障链路 |

## 5. 风险与依赖

### 主要风险

1. 已打包 app 签名信任链不完整，可能导致 Gatekeeper/TCC 不稳定。
2. 多入口启动（CLI/UI/daemon）环境变量注入路径不一致。
3. 文档候选路径与代码候选路径偏差导致运维误判。

### 关键依赖

1. 配置系统支持新增 `screenObserver.binaryPath`。
2. 前端配置页与诊断页联动改造。
3. 如需分发级稳定，需苹果开发者签名与公证流程支持。

## 7. 对外沟通口径（简版）

- 本次故障主因是“二进制解析路径不命中”，不是单纯授权弹窗问题。
- 已有签名打包成果有效，但安装落位与 resolver 白名单不一致，导致重启后仍失败。
- 修复策略为“先止血恢复，再结构化持久化改造，再发布链路标准化”。
