# 多媒体灰度与回滚预案（MM-011）

- 日期：2026-03-04
- 适用范围：Feishu / DingTalk / WeCom 入站多模态分发注入层
- 目标：在不重构插件的情况下，支持按渠道灰度与一键回滚到纯文本链路

## 1. 开关定义

环境变量：`OPENACOSMI_MULTIMODAL_CHANNELS`

语义：

- `all` / `*` / 空值：全部渠道启用多模态（默认）
- `none` / `off` / `false` / `disabled` / `0`：全部渠道禁用多模态（全量回滚）
- `feishu,dingtalk,wecom`：仅启用指定渠道（灰度发布）
- 别名支持：`lark -> feishu`，`wechatwork/workweixin -> wecom`

说明：

- 开关在进程启动时读取。
- 未启用多模态的渠道将自动回退到 `DispatchFunc(text)`，不影响文本消息通路。

## 2. 推荐灰度顺序

1. `OPENACOSMI_MULTIMODAL_CHANNELS=feishu`
2. 稳定后切到 `feishu,dingtalk`
3. 最后切到 `all`

## 3. 一键回滚

紧急回滚到文本链路：

```bash
export OPENACOSMI_MULTIMODAL_CHANNELS=none
# 重启 gateway
```

预期行为：

- 三个渠道不再注入 `DispatchMultimodalFunc`
- 消息仍走文本 dispatch，链路可用性优先

## 4. 验证步骤

1. 启动日志应出现：
   - 启用：`gateway: multimodal dispatch enabled`
   - 回退：`gateway: multimodal dispatch disabled, fallback to text dispatch`
2. 运行回归：

```bash
GOCACHE=/tmp/go-build GOTMPDIR=/tmp make test-multimedia
```

3. 渠道实机抽样：
   - 文本消息必须可正常收发
   - 关闭多模态时，附件应降级为文本占位/提示，不应出现管道阻塞

## 5. 风险与兜底

- 风险：运维误填开关值。
- 兜底：解析策略采用 fail-open（非法值回到全开），避免因拼写问题造成全量不可用。
- 真正回滚请使用显式值：`none`。
