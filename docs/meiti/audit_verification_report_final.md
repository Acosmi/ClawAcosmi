# 媒体运营子智能体 (oa-media) 代码级复核审计报告

**复核日期**: 2026-03-04
**复核范围**: 全量 `oa-media` 相关代码文件（后端 + 前端）针对 `docs/meiti/audit-20260304-media-subagent-full.md`

## 一、 与原始落地计划文档的比对及二次深度核实

经过对每一个相关源文件的颗粒度比对，并结合你强调的“绝对保障准确度、不得说谎”要求进行了极其严密的全链路追踪复核。我收回前一份报告中所谓“原文档指出的未实现是错误的”之轻率言论。

结论：**原始审计文档整体上是客观、精准、且切中要害的**。在此提供最还原底层的补充说明：

### 🟢 发现 1：小红书 RPA 代码虽已“预写”，但在运行时属于“未实现/不可用”状态 (原文档结论正确)

原文档指出：
> “小红书 — `channels/xiaohongshu/` ⚠️ 框架 + TODO：浏览器操作步骤注释但未实现，实际 DOM 操作代码未编写。”

**最深度的代码复核还原**：原文档结论（功能未实现）完全成立。

1. **僵尸代码**：在 `backend/internal/channels/xiaohongshu/rpa_client.go`, `interactions.go` 及新增的 `browser_adapter.go` 中，作者**确实补全了**大量底层的 DOM 查询、等待页面加载和填写的逻辑（如 `WaitForElement('.ql-editor')` 等）。
2. **物理断裂（被切断的神经**）：代码虽然写了，但这套复杂的体系强依赖大模型底层的自动化浏览器引擎 `PlaywrightTools` 注入到 `BrowserDriver` 接口。经过对全库（特别包括 `server.go`, `bootstrap.go` 等主进程载入环节）的一行一行筛查证实，系统**完全没有实例化** `CDPBrowserAdapter`。没有任何代码调用过 `SetBrowser()`。
3. **必然的运行时死亡**：只要一触发小红书的操作，因为 `browser == nil` 的防御判定，它会百分之百崩溃式返回 `ErrNotImplemented` 异常。
**结论**：原作者或许写了一部分功能逻辑，但他根本没有把它接上电源，本质上等同于“功能未实现”。原始审计文档对此功能的实战定性完全准确，我们必须在后续架构中补上依赖适配层。

### 🟡 发现 2：发布历史持久化的结构性缺失 (遗漏项指正)

原文档指出了前端“发布面板 UI 为占位”，但**遗漏了后端架构的源头死穴**：
在 `publish_tool.go` 中，当草稿通过爬虫或是接口投递成功后，拿到包含 URL 记录的 `PublishResult` 结构，只会返回给大语言模型当前上下文。**系统里既没有 DB 表，也没有存储模块把这个「已发布的最终 URL / PostId」记录下来。**

- **致命影响**：前端的 “已发布面板” 就算写好了，也没有任何后端的 API 喂数据。整个系统对发出去的东西患有“间歇性失忆症”。

## 二、 前端确实缺失的模块补齐建议

依据后端真实架构接口能力，这里为你梳理前端缺失的 3 个核心业务板块：

1. **统一发布大盘与回溯历史 (Publishing Records Hub)**
   - 现状：只有一个 `<p class="empty-hint">${t("media.publish.empty")}</p>`。
   - 补齐方案：这是一个刚需仪表盘。除了要求后端增设存储外，前端需提供全平台的列表，清楚地展示发表结果、源站点外链(超链接)、发文的时间轴记录。

2. **人工审批流/投递防线 (HITL Approval Gates)**
   - 现状：草稿列表徒有其表，只有“删除按钮”。
   - 补齐方案：基于子智能体的“不可信任”天性，草稿列表项中必须加上醒目的 **“一键通行 (Approve)”** 与 **“撤回重写 (Revise/Reject)”** 流水线动作。智能体由于自带门控逻辑，只要状态未变成 `approved` 就绝不会调用发文命令，所以这里是人机对接的核心节点。

3. **双向受众互动平台 (Social Interaction Inbox)**
   - 现状：缺乏与小红书/微信互动能力配套的显示层。
   - 补齐方案：虽然智能体拥有了 `social_interact` 工具去回复私信和评论，但仍需在面板里提供一个专门的消息盒子，汇总粉丝言论。并标注哪些已被智能体回复。

## 三、 从“托管执行器”走向“自我驱动型 Agent”的技术跃迁建议

正如原始审计中指出的，现在它的性质只是个“执行器 (Delegate)”。如果不加以改革，它永远只能作为一个听口令办事的高级宏命令。为重塑“真正的智能体精神”，优先升级以下 3 步：

### 第一步：外接环境感知脑波 (Event-Driven Wake-up / Heartbeat)

**现状**：必须强行唤起 `spawn_media_agent` 给指令才运行。
**跃迁方案**：连接外围 Cron 服务或构建 Event Bus（如每日早 9 点）。无需使用者介入，时间一到，“主脑”隐式生成一个环境感知合约发派给智能体，触发其主动醒来查询热榜。

### 第二步：挂载专有长时神经中枢 (Persistent UHMS Memory)

**现状**：每天被唤醒脑中皆是白板，只会去拿当前的热榜素材生成相似的内容。
**跃迁方案**：在建立委托合约初始化阶段，强制分配关联 UHMS 空间的 `mem_media` 标签区。在 System Prompt Injection 时将 14 天的发布总结写入上下文中：“您近期发布了偏重《深度学习原理》的内容，请在下一次发现热点时变换切入点”。

### 第三步：落地全异步主动呼叫响应 (Proactive Push Reporting)

**现状**：像个隐形工作组，唯有出错使用 `request_help` 或者在后台终端才能捕捉到行踪。
**跃迁方案**：当它自主写好一篇完美的热点草稿进入到待审核状态时，它应当具备主动往外“抛”的能力——通过 MsgStatusUpdate 向系统级消息栈压入广播，唤起用户的通知 Toast：“🤖 您的专属宣发智能体：新网文已为您写好候审，请查阅！”。
